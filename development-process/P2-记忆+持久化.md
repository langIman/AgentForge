### 2. 存储层 (`src/storage/`)

#### database.py — SQLite引擎

```python
engine = create_async_engine("sqlite+aiosqlite:///agentforge.db")
SessionLocal = async_sessionmaker(engine)
```

- 使用 SQLAlchemy 2.0 async 驱动
- `init_db()` 调用 `create_all()` 自动建表，无需Alembic迁移
- **升级路径**：只需改连接串 + `pip install asyncpg` 即可切换到PostgreSQL

#### models.py — ORM模型

**Task模型**：

| 字段 | 类型 | 说明 |
|------|------|------|
| id | Integer PK | 自增主键 |
| subject | String(500) | 任务标题 |
| description | Text | 详细描述 |
| status | String(20) | pending / in_progress / completed |
| owner | String(100) | 负责人（P3多Agent认领用） |
| worktree | String(100) | 关联的worktree（P4用） |
| blocked_by | JSON | 依赖的上游任务ID列表 |
| blocks | JSON | 阻塞的下游任务ID列表 |
| created_at / updated_at | Float | 时间戳 |

**Transcript模型**：

| 字段 | 说明 |
|------|------|
| session_id | 关联的会话ID |
| messages | JSON，压缩前的完整消息列表 |
| summary | 压缩时生成的摘要 |

#### repository.py — TaskRepository

核心方法：

| 方法 | 功能 |
|------|------|
| `create(subject, description, blocked_by)` | 创建任务，自动维护双向依赖 |
| `get(task_id)` | 获取单个任务 |
| `update(task_id, **kwargs)` | 更新字段，**completed时自动清除下游依赖** |
| `list_claimable()` | 列出可认领任务（pending + 无owner + 无阻塞） |
| `claim(task_id, owner)` | 认领任务 |

**依赖图自动解锁**是最关键的逻辑：

```
任务A (blocked_by: []) ──完成──→ 自动从任务B的blocked_by中移除A
任务B (blocked_by: [A])         任务B (blocked_by: []) → 可执行
任务C (blocked_by: [A, B])     任务C (blocked_by: [B]) → 仍等B
```

### 3. 记忆层 (`src/memory/`)

#### compressor.py — 三层上下文压缩

| 层 | 名称 | 触发时机 | 做了什么 |
|----|------|---------|---------|
| 层1 | `micro_compact` | **每轮**（pre_process中） | 截断旧ToolMessage为`[cleared: tool_name]`，保留最近3个 |
| 层2 | `auto_compact` | token > 80K 阈值 | 存档到SQLite → LLM生成摘要 → 用摘要替换全部历史 |
| 层3 | `manual_compact` | Agent调用compact工具 | 与层2相同，但不受阈值限制 |

**为什么需要三层？**

- 层1解决工具输出膨胀（一次bash输出可能几千字符，但历史中只需知道"执行过bash"）
- 层2解决长对话溢出（80K token时自动压缩，防止超出模型窗口）
- 层3给Agent自主权（意识到上下文过长时可主动压缩，不必等阈值）

**token估算**：`len(json_string) / 4`，不精确但足够做阈值判断。

#### transcript.py — 对话存档

auto_compact触发时，先将完整消息存入Transcript表，再生成摘要替换。
这样即使压缩了，原始对话也可以回溯。

### 4. 新增工具

#### compact工具 (`src/tools/compact.py`)

Agent调用后返回标记 `[COMPACT_REQUESTED]`，pre_process检测到这个标记后执行实际压缩。

为什么不在工具内直接压缩？因为工具无法修改State中的messages列表，只能通过节点来修改。

#### task CRUD工具 (`src/tools/task.py`)

4个工具对应TaskRepository的4个操作：

| 工具 | 功能 | 特殊逻辑 |
|------|------|---------|
| `task_create` | 创建任务 | 支持指定blocked_by依赖 |
| `task_update` | 更新状态/描述/负责人 | completed时自动解锁下游 |
| `task_list` | 列出任务 | 可按状态过滤 |
| `task_get` | 查看详情 | 显示依赖关系 |

**与P1的TodoManager的区别**：

| | TodoManager (P1) | TaskRepository (P2) |
|--|-----------------|-------------------|
| 存储 | 内存 | SQLite |
| 跨会话 | 不保存 | 持久化 |
| 依赖图 | 无 | blocked_by / blocks |
| 多Agent | 不支持 | owner + claim机制 |
| 用途 | 当前工作的轻量待办 | 项目级的持久任务 |

两者共存，互不替代。

### 5. nodes.py 修改 — pre_process加入压缩

P1的pre_process是空操作（`return {}`），P2改为：

```
pre_process:
  1. 检测是否有手动compact请求（扫描最近ToolMessage）
  2. micro_compact — 截断旧工具输出
  3. estimate_tokens — 计算当前token数
  4. 如果超阈值或有手动请求 → auto_compact（存档+摘要+替换）
  5. 返回更新后的messages和token_count
```

**不改图拓扑** — 这是核心原则。所有新逻辑塞进现有节点内部。

## 关键设计决策

### 为什么用SQLAlchemy而不是直接sqlite3？

与plan.md代码100%兼容。将来换PostgreSQL只需改连接串，不改任何业务代码。

### 为什么压缩在pre_process而不是post_process？

pre_process在LLM调用之前执行。压缩必须发生在调用前，否则被压缩的内容已经发送给了LLM。

### 为什么compact工具不直接压缩？

LangGraph中tool的返回值只能作为ToolMessage追加到消息列表。要替换/删除消息，必须通过节点（node）修改State。所以compact工具只设标记，pre_process做实际操作。

### 为什么TodoManager和TaskRepository共存？

职责不同：
- TodoManager是"便利贴"，跟踪当前工作步骤，每轮可以完全重写
- TaskRepository是"项目管理板"，跨会话持久化，有依赖图，支持多Agent协作
